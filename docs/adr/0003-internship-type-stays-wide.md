# Internship type stays a wide struct

`src/lib/types.ts` `Internship` carries 35+ fields across five concerns (identity, discovery metadata, user state, scoring, link health, salary). Splitting into `{identity, scoring, userState, health, salary}` would make boundaries explicit and concentrate ownership — but every existing caller currently knows all five concerns and would need to coordinate across the split with no behavior change. **Deferred** as a design-debt marker. Real bugs from the wide struct (e.g., `revalidateLinks` upserting every column and clobbering user state, `refreshOnRediscovery` overwriting `link`) have all been mitigated at the SQL layer via narrow per-id UPDATEs. Split when an integration pressure forces it — e.g., extracting notifications into a separate process, or an admin export that should never see user state.
